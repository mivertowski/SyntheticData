# DataSynth Output Viewer

A web-based viewer for exploring synthetic data generated by [SyntheticData](../README.md) (datasynth). It provides tabbed views over journal entries, master data, fraud & anomaly labels, trial balance, general/auxiliary ledgers, subledgers, and an interactive graph (in-memory or Neo4j).

## Features

| View | Description |
|------|-------------|
| **Journal (JEC / FEC)** | Browse journal entries (CSV/JSON) or French *Fichier des Écritures Comptables* (FEC) with search and pagination. |
| **Master Data** | Vendors, customers, materials, fixed assets, and employees from `master_data/`. |
| **Fraud & Anomalies** | Anomaly/fraud labels with charts (by category, type, severity), summary stats, and drill-down to concerned journal lines. |
| **Trial Balance** | Trial balance data with account-level totals. |
| **General Ledger** | General ledger view. |
| **Auxiliary Ledger** | Auxiliary ledger (e.g. vendor/customer sub-accounts). |
| **Subledger** | AR/AP invoices, fixed-asset records, inventory positions, and subledger reconciliation. |
| **Graph (Neo4j)** | Interactive force-directed graph: **in-memory** (built from journal entries) or **Neo4j** (connect to a running instance). |

Additional behavior:

- **Load data**: Point the app at a different base URL or path (e.g. `/data`, or a server serving your output folder). The choice is persisted in `localStorage`.
- **Theme**: Light, dark, or system. Persisted in `localStorage`.
- **Responsive**: Tables support paging and scroll; graph supports zoom/pan and node filtering (whitelist, account depth).

## Requirements

- **Node.js** 18+ and npm (or compatible package manager).
- Generated output from `datasynth-data generate` (or equivalent). The viewer expects the same [output structure](#data-source) as the generator.

## Quick Start

### 1. Install and run (development)

```bash
cd datasynth-output-viewer
npm install
npm run dev
```

Open the URL shown (e.g. `http://localhost:5173`). By default the app loads data from `/data` (see [Data source](#data-source)).

### 2. Use your own output directory

**Option A — Copy output into the app (for local dev):**

```bash
# From repo root, after generating:
datasynth-data generate --config config.yaml --output ./output

# Then in datasynth-output-viewer:
OUTPUT_DIR=../output npm run load-data
npm run dev
```

`load-data` copies the contents of `OUTPUT_DIR` (default `./output`) into `public/data`, so the dev server serves it at `/data`.

**Option B — Change base at runtime:**

1. Run the app (`npm run dev` or after build).
2. Click **Load data** in the header.
3. Enter a base path (e.g. `/data`) or full URL (e.g. `https://example.com/my-output`). Relative paths are resolved from the app origin.

### 3. Production build

```bash
npm run build
npm run preview   # serve dist/ locally
```

For production, serve the built app (e.g. via nginx or a static host) and either:

- Serve the generator output under a path that matches the configured base (e.g. `/data`), or
- Have users set **Load data** to the correct URL after opening the app.

## Data source

The viewer fetches data from a **base URL** (default `/data`). All requests are relative to this base, e.g.:

- `{base}/journal_entries.json` or `{base}/journal_entries.csv`
- `{base}/fec.csv`
- `{base}/master_data/vendors.json`, `customers.json`, etc.
- `{base}/labels/anomaly_labels.json`, `anomaly_labels_summary.json`
- `{base}/chart_of_accounts.json`
- `{base}/document_flows/*.json`
- `{base}/subledger/*.json`
- `{base}/balance_validation.json`, `{base}/balance/subledger_reconciliation.json`
- `{base}/generation_statistics.json`, `{base}/run_manifest.json`

You can override the default base:

- **Build time**: set `VITE_DATA_BASE` in `.env` (e.g. `VITE_DATA_BASE=/api/output`).
- **Runtime**: use **Load data** in the UI; the value is stored in `localStorage` under `datasynth-viewer-data-base`.

## Scripts

| Script | Description |
|--------|-------------|
| `npm run dev` | Start Vite dev server. |
| `npm run load-data` | Copy `OUTPUT_DIR` (default `./output`) into `public/data`. |
| `npm run dev:with-data` | `load-data` then `dev`. |
| `npm run build` | TypeScript build + Vite production build. |
| `npm run preview` | Serve `dist/` (after build). |
| `npm run preview:with-data` | `load-data` → `build` → `preview`. |

Environment:

- `OUTPUT_DIR` — directory to copy into `public/data` when running `load-data` (default: `./output`).

## Graph view

- **In-memory**: Loads journal entries and chart of accounts, then builds a graph of accounts (nodes) and flows (edges) from debits/credits. Options: whitelist (account codes), include auxiliary accounts, max account hierarchy depth, and row limit.
- **Neo4j**: Connect to a running Neo4j instance (URI, user, password). Loads graph data via Cypher; optional whitelist filter. Connection details can be stored in `localStorage`.

## Tech stack

- **React** 19, **TypeScript**, **Vite**
- **Recharts** (fraud/anomaly charts)
- **force-graph** (graph visualization)
- **PapaParse** (CSV)
- **neo4j-driver** (optional Neo4j backend)
