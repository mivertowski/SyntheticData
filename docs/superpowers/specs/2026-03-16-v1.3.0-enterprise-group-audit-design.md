# v1.3.0 Design Spec: Enterprise Group Audit Simulation

**Date**: 2026-03-16
**Version**: 1.3.0
**Status**: Draft (Rev 2 â€” post spec-review fixes)
**Scope**: All findings from the enterprise group audit readiness evaluation (30 items + 2 additions from review)

## Overview

v1.3.0 transforms DataSynth from a transactional data generator into a comprehensive enterprise group audit simulation platform. The release addresses 32 items (30 original findings + 2 additions from spec review: ISA 540 Accounting Estimates and GroupStructure ownership model) across standards coverage, data coherence, output completeness, and evaluation depth â€” organized into 5 implementation tiers that each leave the codebase in a releasable state.

**Primary use case**: Simulate an entire enterprise group audit from both internal and external perspectives, following ISA, IFRS, US GAAP, and local regulations.

## Architecture Principles

1. **Existing patterns preserved**: All new models follow the `datasynth-core/src/models/` â†’ `datasynth-generators/src/` â†’ `datasynth-runtime` orchestrator â†’ `datasynth-cli` output writer pipeline.
2. **Workspace version**: All crates bump to 1.3.0. Workspace dependency versions in root `Cargo.toml` must match (lesson from v1.2.0 publish fix).
3. **Config-driven**: Every new feature is gated by a config section with `enabled: bool`. Existing configs remain backward-compatible.
4. **Framework-aware**: All accounting models support dual-framework (IFRS + US GAAP) where standards differ. Framework selection flows from `FrameworkSettings`.
5. **GL-integrated**: Every new accounting standard that produces financial impacts must generate `JournalEntry` records that flow into the trial balance and financial statements.

### 0.2 Subledger: Payment Application Reduces amount_remaining

**Problem**: `APInvoice.amount_remaining` and `ARInvoice.amount_remaining` are set at invoice creation and never reduced by payments. This causes systematic overstatement in subledger-to-GL reconciliation.

**Solution**: After document flow generation, iterate all payments and receipts, match them to invoices by reference, and subtract from `amount_remaining`.

**Files changed**:
- `crates/datasynth-generators/src/subledger/document_flow_linker.rs` â€” add `apply_payment_settlements()` and `apply_receipt_settlements()` methods
- `crates/datasynth-runtime/src/enhanced_orchestrator.rs` â€” call settlement logic after subledger linking

**Logic**:
```
for each Payment in p2p_chains:
    find APInvoice where invoice_reference matches payment.invoice_id
    ap_invoice.amount_remaining = max(0, ap_invoice.amount_remaining - payment.amount)

for each Payment in o2c_chains where payment_type == ArReceipt:
    find ARInvoice where invoice_reference matches payment.invoice_id
    ar_invoice.amount_remaining = max(0, ar_invoice.amount_remaining - payment.amount)
```

Handles partial payments (multiple payments against one invoice) and overpayments (clamp to zero with logging).

**Test**: Integration test verifying that after settlement, subledger totals match GL control account balances within tolerance.

### 0.3 IC Eliminations to GL

**Problem**: `EliminationEntry` records are generated as domain objects but never converted to `JournalEntry` records. Consolidated financial statements include IC transactions without eliminations.

**Solution**: Add `to_journal_entry()` conversion on `EliminationEntry` and append converted JEs to the main entries vector.

**Files changed**:
- `crates/datasynth-generators/src/intercompany/elimination_generator.rs` â€” add `impl EliminationEntry { pub fn to_journal_entry(&self) -> JournalEntry }`
- `crates/datasynth-runtime/src/enhanced_orchestrator.rs` â€” after elimination generation, convert and append

**Design details**:
- Each elimination entry already has debit/credit account codes and amounts
- Converted JE gets `source: "CONSOLIDATION"`, `document_type: "ELIMINATION"`
- `is_elimination: bool` flag added to `JournalEntryHeader` (consistent with existing flags: `is_fraud`, `is_anomaly`, `sox_relevant`, `sod_violation` which all live on `JournalEntryHeader`)
- Conversion logic placed in `datasynth-generators/src/intercompany/elimination_generator.rs` as a standalone `elimination_to_journal_entry()` function (not as a method on the model â€” consistent with codebase pattern where `datasynth-core` models are pure data structures and conversion logic lives in `datasynth-generators`)
- Reference back to `EliminationEntry.id` for audit trail

**Test**: Verify elimination JEs balance (debits = credits), verify consolidated trial balance reflects eliminations.

### 0.4 Wire CloseEngine into Orchestrator

**Problem**: The `CloseEngine` in `crates/datasynth-generators/src/period_close/close_engine.rs` has full task dependency management but is never called by the orchestrator. Period-close entries don't flow through.

**Solution**: Add `phase_period_close()` to the orchestrator, wired between anomaly injection and financial reporting.

**Files changed**:
- `crates/datasynth-runtime/src/enhanced_orchestrator.rs` â€” new phase method

**Handler wiring**:
| Close Task | Handler | Source |
|---|---|---|
| Depreciation | `DepreciationGenerator::generate()` | Existing generator |
| Accruals | `AccrualsGenerator::generate()` | Existing generator |
| FX Revaluation | `FxRateService::revalue_monetary_items()` | Existing service |
| Tax Provision | Stub in Tier 0 (simple: statutory_rate * pre_tax_income â†’ DR Tax Expense / CR Tax Payable), replaced by full deferred tax engine in Tier 1.3. Stub produces valid JEs so the interface is stable across tiers. | New |
| Income Statement Close | New closing entry generator | New (DR Revenue, CR Income Summary; DR Income Summary, CR Expenses; DR/CR Income Summary â†’ Retained Earnings) |
| Retained Earnings Rollforward | Derived from closing entries | New |

All handler outputs are `Vec<JournalEntry>` appended to the main entries vector.

**Test**: Verify close engine executes all tasks in dependency order. Verify retained earnings equals prior RE + net income - dividends.

### 0.5 Opening Balances as JEs

**Problem**: `GeneratedOpeningBalance.balances` are generated but not converted to JEs. The `RunningBalanceTracker` starts from zero, causing incorrect balance sheet state until sufficient transactions accumulate.

**Solution**: Convert opening balances to JEs and prepend to the entries vector.

**Files changed**:
- `crates/datasynth-generators/src/balance/opening_balance.rs` â€” add `to_journal_entries(&self, coa: &ChartOfAccounts) -> Vec<JournalEntry>`
- `crates/datasynth-runtime/src/enhanced_orchestrator.rs` â€” call conversion, prepend to entries

**Design details**:
- Each opening balance becomes a JE with `source: "OPENING_BALANCE"`, `date: period_start_date`
- For contra-asset accounts (accumulated depreciation), use `AccountType` from CoA lookup to determine correct debit/credit side
- Opening balance JEs use a balanced multi-line entry: assets on debit, liabilities + equity on credit

**Test**: Verify balance tracker reports correct balances from period start. Verify BS equation holds for opening balance JEs.

**New generator** in `datasynth-generators/src/standards/business_combination_generator.rs`:
- Generates 1-3 acquisitions per multi-year run
- Creates PPA with realistic step-ups: PP&E (10-25%), customer relationships (new intangible), trade names (new intangible), technology (new intangible)
- Computes goodwill = consideration - net FV of identifiable assets
- Day 1 JEs: DR Assets at FV, DR Goodwill, CR Liabilities at FV, CR Cash/Shares
- Subsequent: amortization of finite-lived acquired intangibles, annual goodwill impairment test

**Reconciliation with existing goodwill model**: The existing `EliminationType::Goodwill` in `elimination.rs` and `create_investment_equity_elimination()` handle goodwill at the consolidation level. The new `BusinessCombination` model is the **source of truth** for goodwill amounts â€” it computes goodwill via PPA. The elimination generator's `Goodwill` type is updated to consume `BusinessCombination.goodwill` rather than computing its own value. Flow: `BusinessCombinationGenerator` â†’ `BusinessCombination.goodwill` â†’ `EliminationGenerator` reads this value for `EliminationType::Goodwill` entries.

**Multi-period architecture**: Business combination effects span multiple periods. The generator pre-computes the full schedule at generation time:
1. Day 1: acquisition JEs (recorded at acquisition date)
2. Subsequent periods: amortization JEs for finite-lived intangibles (pre-computed schedule stored in `FairValueAdjustment.useful_life_years`)
3. Annual: goodwill impairment test records (generated for each year-end after acquisition)

All JEs are generated upfront with their respective dates and appended to the entries vector during the business combination phase. The orchestrator does not need per-period state â€” the generator produces the complete multi-period JE set.

**Config section**:
```yaml

    enabled: true
    acquisition_count: 2
    avg_goodwill_percent: 0.35
```

**Output**: `accounting_standards/business_combinations.json`, `accounting_standards/purchase_price_allocations.json`

### A.1 ISA 540 â€” Accounting Estimates (Tier 3 addition)
